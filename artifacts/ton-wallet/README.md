# TON Testnet Wallet

Self-custodial кошелёк для TON testnet без собственного backend.

## Запуск

```bash
pnpm install
pnpm --filter @workspace/ton-wallet run dev
```

Приложение доступно на `http://localhost:<PORT>/`

## Тесты

```bash
pnpm --filter @workspace/ton-wallet run test
```

## Стек

- **React + Vite + TypeScript** — SPA, никаких серверных зависимостей
- **@ton/ton, @ton/crypto, @ton/core** — официальные TON библиотеки
- **TON Testnet API** — `testnet.toncenter.com` (публичный JSON-RPC)
- **localStorage** — хранение мнемоники и истории адресов
- **Vitest** — unit-тесты

## Архитектура и компромиссы

### Принципиальные решения

#### 1. Без backend — полностью в браузере
Всё взаимодействие с блокчейном идёт напрямую с `testnet.toncenter.com`:
- Получение баланса: `GET /getBalance`
- Получение транзакций: `GET /getTransactions`
- Отправка: через `@ton/ton` `TonClient`

**Компромисс:** toncenter.com — публичный API с лимитами (1 req/s без API ключа). В продакшне нужен собственный API ключ или нода.

#### 2. Хранение мнемоники в localStorage
Мнемоника хранится в `localStorage` в открытом виде.

**Почему:** ТЗ прямо указывает "production-grade безопасность не требуется".

**Что нужно в продакшне:**
- Шифрование мнемоники паролем пользователя (AES-GCM через WebCrypto API)
- Опционально — хранение только зашифрованного blob
- Рассмотреть hardware wallet интеграцию (Ledger)

#### 3. WalletV4R2 как тип кошелька
Используется `WalletContractV4` — наиболее распространённый тип в экосистеме TON на момент написания.

**Компромисс:** Новый WalletV5 поддерживает gasless транзакции, но менее распространён в testnet инструментах.

#### 4. Защита от адрес-подмены (Address Poisoning)
Реализованы три слоя защиты:

1. **Levenshtein distance** — если адрес отличается от ранее использованного на ≤3 символа, выдаётся `danger`-предупреждение и отправка блокируется.
2. **Отправка на себя** — `danger`, отправка блокирована.
3. **Новый адрес** — `info`-предупреждение, отправка не блокируется.
4. **95% баланса** — `warning` о газе.
5. **>100 TON** — `warning` о крупной сумме.

**Логика блокировки:** при наличии хоть одного `danger`-предупреждения кнопка подтверждения деактивируется. `warning` и `info` показываются, но не блокируют.

**Реалистичные сценарии подмены:**
- Атакующий отправляет 0.001 TON с адреса, похожего на адрес получателя — пользователь копирует адрес из истории транзакций не проверяя → защита обнаруживает схожесть
- Пользователь вводит адрес вручную с опечаткой → валидация формата
- Пользователь указывает свой адрес → явное предупреждение

#### 5. UX при долгом ответе сети
После нажатия "Подтвердить":
1. Экран переходит в состояние "Отправка..." с анимацией
2. Транзакция отправляется (`contract.send()`)
3. После успеха/ошибки — отдельный экран с результатом
4. Кошелёк автоматически обновляется через 3 сек (время на индексацию)

### Что не реализовано (и почему)

| Фича | Причина |
|------|---------|
| Шифрование мнемоники паролем | Вне скоупа ТЗ |
| WalletV5 | Менее зрелая поддержка в testnet |
| Поддержка нескольких аккаунтов | Усложнение UX без необходимости |
| Jetton (токены) | Не упомянуто в ТЗ |
| NFT | Не упомянуто в ТЗ |

### Дальнейшие улучшения

1. **Безопасность:** шифровать мнемонику паролем через WebCrypto
2. **Rate limiting:** собственный API ключ toncenter или собственная нода
3. **UX:** WebSocket для real-time обновлений баланса
4. **Multi-account:** поддержка нескольких кошельков
5. **DeepLinks:** `ton://transfer/ADDRESS?amount=X` для QR-кодов
6. **Фиат-эквивалент:** показывать баланс в USD/EUR через CoinGecko
7. **Passphrase для мнемоники:** BIP39 passphrase как дополнительный фактор
8. **Batch транзакции:** отправка нескольким получателям за раз (WalletV5)
