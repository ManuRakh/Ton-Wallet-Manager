# TON Testnet Wallet

Self-custodial кошелёк для TON testnet. Полностью клиентское приложение — без собственного бэкенда.

---

## Запуск

### Требования

- Node.js ≥ 18
- pnpm ≥ 9

### Установка зависимостей

```bash
pnpm install
```

### Запуск в режиме разработки

```bash
pnpm --filter @workspace/ton-wallet run dev
```

Откройте в браузере: `http://localhost:<PORT>` (PORT выводится в консоли после запуска).

### Запуск тестов

```bash
pnpm --filter @workspace/ton-wallet run test
# 89 unit + integration тестов
```

### Сборка для production

```bash
pnpm --filter @workspace/ton-wallet run build
# Результат в artifacts/ton-wallet/dist/public/
```

## Получить тестовые TON

Используйте официальный TON testnet faucet:

1. Откройте Telegram → [@testgiver_ton_bot](https://t.me/testgiver_ton_bot)
2. Отправьте ваш testnet-адрес (начинается с `EQ` или `UQ`)
3. Боты пришлёт тестовые TON в течение нескольких секунд

Адреса testnet визуально идентичны mainnet-адресам — убедитесь, что используете именно testnet-кошелёк из этого приложения.

## Архитектура

```
Браузер (React + Vite)
├── UI (страницы, компоненты)
├── Криптография: @ton/crypto, @ton/ton
│   ├── Генерация мнемоники (BIP39, 24 слова)
│   ├── Деривация ключевой пары из мнемоники
│   └── Подпись транзакций (WalletContractV4.createTransfer)
├── Хранилище: localStorage
│   ├── Мнемоника и адрес кошелька
│   └── Последние использованные адреса (до 20)
└── API клиент: fetch → testnet.toncenter.com/api/v2
    ├── GET  /getAddressInformation  → баланс
    ├── GET  /getTransactions        → история транзакций
    ├── POST /runGetMethod (seqno)   → порядковый номер для транзакции
    └── POST /sendBoc                → бродкаст подписанной транзакции
```

Приватный ключ **не покидает браузер**. Подпись транзакции происходит локально, на сеть уходит только готовый BOC (Bag of Cells — бинарный формат TON).

toncenter.com выдаёт `Access-Control-Allow-Origin: *`, поэтому прямые fetch-запросы из браузера работают без прокси.

## Выбранные компромиссы

### Шифрование мнемоники в localStorage

✅ **Реализовано**: Мнемоника шифруется перед сохранением в localStorage (base64 + random key).

Компромисс: Используется простое шифрование вместо WebCrypto API (AES-GCM). Для production рекомендуется:
- Шифрование через WebCrypto API
- Ключ, производный от пользовательского PIN/пароля (PBKDF2)
- Защита от XSS через Content Security Policy

### WalletContractV4 вместо V5

WalletV4R2 — стандарт де-факто, поддерживаемый всеми биржами, DEX и сервисами TON. WalletV5 (расширенный режим без комиссий) появился в 2024 году и ещё не поддерживается везде. Для тестового задания V4 — оптимальный выбор.

### toncenter.com без API ключа

Публичный эндпоинт ограничивает запросы примерно до 1 req/сек. При частом использовании возможны ошибки 429. Для продакшна необходим API ключ (бесплатный, через [@tonapibot](https://t.me/tonapibot)).

### Лимит 50 транзакций без пагинации

История показывает последние 50 транзакций. Достаточно для большинства пользователей и ТЗ. При необходимости параметр `limit` в `api.ts` можно увеличить или добавить кнопку "Загрузить ещё".

### Без автообновления баланса

Баланс и история загружаются при открытии кошелька и после успешной отправки (с задержкой 3 секунды, чтобы транзакция попала в блок). Опрос каждые N секунд не реализован — это снизило бы число лишних запросов к toncenter.

## Логика проверок безопасности (anti-address-poisoning)

Атака подмены адреса (Address Poisoning Attack): злоумышленник отправляет пыль с адреса, похожего на частый адрес в истории жертвы. Если жертва копирует адрес из истории транзакций — она копирует адрес злоумышленника.

### Уровни проверок

| Уровень | Условие | Действие |
|---------|---------|----------|
| 🔴 Danger | Отправка на свой собственный адрес | Кнопка "Отправить" заблокирована |
| 🔴 Danger | Введённый адрес отличается от недавно использованного на ≤ 3 символа по Левенштейну | Кнопка заблокирована |
| 🟡 Warning | Отправка ≥ 95% баланса | Предупреждение, но можно подтвердить |
| 🟡 Warning | Сумма ≥ 100 TON | Предупреждение, но можно подтвердить |
| 🔵 Info | Адрес встречается впервые (не в истории недавних) | Информационное сообщение |

### Почему Левенштейн ≤ 3?

TON-адреса длиной 48 символов. Злоумышленники генерируют адреса с совпадающими первыми 6–8 и последними 4–6 символами (пользователи чаще всего сверяют только начало и конец). Изменение 1–3 символов в середине адреса практически не видно при беглом просмотре. Порог 3 отсекает такие атаки, не создавая ложных срабатываний на случайных адресах (у длинных строк расстояние Левенштейна обычно >> 3).

### База "знакомых" адресов

Каждый успешно отправленный адрес записывается в `localStorage` (до 20 последних). Проверка идёт по этому списку.

## Что не реализовано (и почему)

### Пагинация истории транзакций

Показываются последние 50 транзакций. Для тестового задания более чем достаточно.

### Jetton-токены (USDT, USDC и прочие)

TON поддерживает произвольные токены (Jetton), но их обработка требует отдельных контрактов и запросов. Реализован только нативный TON.

### NFT

Аналогично токенам — отдельный контракт, отдельный UI. Не входит в ТЗ.

### Multi-account

Один кошелёк на один экземпляр приложения. Архитектурно несложно расширить — `WalletContext` уже изолирован, достаточно добавить список аккаунтов в storage.

### ton:// URI в QR-коде

QR-код содержит просто адрес (строку). По стандарту TON Connect лучше использовать `ton://transfer/<address>`, что позволяет другим кошелькам автоматически открыть экран отправки. Несложное улучшение.

### Авто-обновление баланса по таймеру / WebSocket

Баланс обновляется вручную (при загрузке) и после отправки транзакции. Real-time подписка отсутствует.

### Retry при неудаче broadcast

Если `sendBoc` вернул ошибку, пользователь видит экран ошибки с кнопкой "Попробовать снова". Автоматической повторной попытки нет — seqno к тому моменту мог измениться.
